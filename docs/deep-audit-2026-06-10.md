### High

**DA-H1 [V] ‚Äî Fixed NPRIs are deducted from the lessee's NRI; the burdened lessor's royalty row stays whole. Contrary to standard Texas division-order practice.**
- Anchors: `src/components/leasehold/leasehold-summary.ts:550-557` (`npriAdjustedNriBeforeOrriRate = safeNriBeforeOrriRate.minus(fixedNpriBurdenRate)`), `:1007-1023` (`netOwnerTractRoyalty = ownerTractRoyalty ‚àí floatingNpriBurdenRate` ‚Äî floating only), `:1113` (pre-WI = NPRI-adjusted NRI ‚àí ORRI). Documented as deliberate in `LANDMAN-MATH-REFERENCE.md` ¬ß7 ("Fixed NPRIs ‚Ä¶ reduce the NRI base before NRI-basis ORRIs and retained WI are calculated").
- What/why: take 100% minerals subject to an outstanding fixed 1/16 NPRI, leased at 1/4: standard division-order allocation is lessor 1/4 ‚àí 1/16 = 0.1875, NPRI 0.0625, lessee NRI 0.75. LANDroid produces lessor 0.25, NPRI 0.0625, lessee NRI 0.6875 (`1 ‚àí 0.25 ‚àí 0.0625`). The transfer-order total balances either way (which is why the variance check can't catch it), but the split is wrong: a fixed NPRI is carved from the mineral estate the lessor leased, and the lessor's royalty bears it to the extent of the royalty. The lessee bargained for NRI = share √ó (1 ‚àí royalty); LANDroid permanently understates every lessee/retained-WI decimal and overstates every burdened lessor's check by exactly the fixed-NPRI decimal. Floating NPRIs are handled correctly (out of royalty, per-slice, clamped, `:1016-1021`).
- Texas-law caveat, flagged for attorney review rather than asserted: when the fixed NPRI **exceeds** the lease royalty (1/4 NPRI under a 1/8 lease), who bears the excess is genuinely contested (subject-to language, warranty). The proposal below preserves LANDroid's current behavior exactly for that overflow case.
- Confidence: high on the common case (royalty ‚â• NPRI); the overflow case is the documented attorney question.
- Proposed change (golden-master-updating, so deliberate and reviewed per the rebuild contract):
  - In the NPRI loop (`leasehold-summary.ts:956-981`), accumulate fixed burdens per-slice into `leaseSlice.fixedNpriBurdenRate` exactly as today, but split each slice's fixed burden into `coveredByRoyalty = min(fixedBurden, remainingSliceRoyalty)` and `excess = fixedBurden ‚àí coveredByRoyalty`.
  - Net the covered part from the owner row (extend `:1007-1023` to subtract `coveredByRoyalty` alongside floating), and subtract only `Œ£ excess` in `calculateOrriBasisRates`' `fixedNpriBurdenRate` input (`:554`, `:1083-1102`) with a new warning flag `fixedNpriExceedsRoyalty` (mirroring `overFloatingNpriBurdened`).
  - Tests: extend `src/components/leasehold/__tests__/leasehold-summary.test.ts` with the 1/16-fixed-under-1/4-lease scenario asserting lessor 0.1875 / NPRI 0.0625 / pre-WI 0.75, plus the overflow scenario asserting today's behavior + the new flag. Regenerate `fixtures/phase-0/demo.leasehold-decimals.json` in the same PR with the delta called out in the PR body, and update `LANDMAN-MATH-REFERENCE.md` ¬ß6.8/¬ß7 with both conventions and the attorney note. Do not ship without the operator hand-checking one tract against a real DO.

## 3. Precision policy recommendation

**Current state.** Internal precision 40 (`decimal.ts:10-13`), storage 24 significant digits via `serialize()` (`:47-53` ‚Äî deliberate, idempotent, error ~1e-24, far below the 1e-9 EPSILON; acceptable and documented), display standard `DISPLAY_PRECISION = 9` via `dualDisplay()` (`fraction-display.ts:150-153`). The sanctioned path is healthy; almost nothing uses it. **Exactly one screen** renders through the sanctioned `FractionDisplay` component (`CurativeView.tsx:886`). The dominant Leasehold formatter is `formatPercent` = 2dp percent = 4 decimal places of the fraction (`LeaseholdView.tsx:110`, ~60 render sites), Desk Map coverage uses 2dp percent (`DeskMapView.tsx:444`), tooltips use 4dp percent / 7dp decimals / 3dp acres (`leasehold-formulas.ts:26,31,36`; `deskmap-formulas.ts:21,24`), and the cascade-factor preview is 6dp (`NodeEditModal.tsx:287`). 4. Seed/demo generators (`seed-test-data.ts:963-971,1355-1365,1394`; `seed-vulcan-mesa.ts:407-483`) ‚Äî float64 + 9dp at seed time; demo-only, but it means demo chains can never exercise >9dp behavior. Low priority; regenerate seeds through Decimal when next touched.
5. `engine/decimal.ts:52` ‚Äî the 24-sig-digit storage cap itself: keep, it is the documented policy.

**The policy (one source of truth):** create `src/engine/display-format.ts` exporting exactly four functions, all built on `DISPLAY_PRECISION`:
- `formatInterestDecimal(v): string` ‚Üí `d(v).toFixed(DISPLAY_PRECISION)` (9dp ‚Äî supersedes every 2/4/6/7/8dp ad-hoc interest formatter);

## 6. Document-storage assessment

**Where it stands.** The registry half is genuinely good for a local-first app: PDF-only originals with magic-byte validation, SHA-256 computed over exactly the stored bytes at write (`document-store.ts:258-273`), metadata-only patch types that exclude blob/hash (`:118-137`), blob-free listing enforced by tests, workspace+dbKey scoping, write-fence on every mutator, and a duplicate-hash badge. The backend-spine record model (`contracts.ts:151-309`) ‚Äî versions, vault objects, extraction runs, citations, anchors, consent-gated cloud OCR ‚Äî is well-designed **and entirely dormant**: zero live callers project or consume it; the strong deterministic packet/manifest/ZIP code (`evidence-vault.ts:804-824`, `packet-archive.ts`) is dead code today.

**Where it fails the "evidence vault" claim.** Three of the ¬ß1 findings are the substance: originals destroyed by routine flows (DA-H6), fixity never re-verified and import-trusted (DA-H7), and no custody/audit events for documents at all ‚Äî title gets a hash chain; document create/rename/retag/delete write nothing anywhere, `updatedAt` is overwritten in place (`document-store.ts:389-393`). Other parallel stores (owner docs / map assets / research imports) carry **no hashes at all** (`evidence-vault.ts:347,397,470` project `contentHash: null`), so cross-store dedup and fixity are impossible. Deletes are hard deletes with no tombstone; the single AI-undo slot is the only net.

**Gap table vs eDiscovery-grade** (verified): Bates/control numbers ‚Äî stub sidecar only (`evidence-vault.ts:885-892`, never persisted/stamped); Concordance/Opticon load files ‚Äî missing (sidecars are JSON); chain-of-custody ‚Äî missing for documents; dedup ‚Äî detection-only (re-upload stores a second blob; `saveDoc` never checks hash); families (parent/attachment) ‚Äî missing (`document_attachments` is doc‚Üîentity, not doc‚Üîdoc); redaction ‚Äî missing; who-did-what ‚Äî missing (no user identity on rows); deterministic manifests ‚Äî strong but unwired (live manifest is filter-order-dependent unhashed JSON).

**Assessment of the three-pane target:** right target, wrong first step. The current two-pane DocumentsView already has saved views, filters, metadata editing, and packet preview (`DocumentsView.tsx:511-944`); what blocks professional use is not the third pane ‚Äî it's that the registry can't even ingest a file directly (upload exists only inside the node-edit modal; the "Inbox" area can't be filled), can't sort columns, previews only in a modal, and sits on the loss/fixity holes above.

**Minimal path (ordered):**
1. Fix DA-H6 + DA-H7 (export completeness, import re-hash, one-time `''` backfill). Without these the vault story is false.
2. Document audit events: reuse the action-layer event shape ‚Äî append `document.*` audit events (create/rename/retag/delete/detach) to the existing hash-chained ledger or a parallel `documentAuditEvents` table (~1 day; gives chain-of-custody and who-did-what in one move, and makes hard-deletes tombstoned as a side effect: keep the row minus blob, `deletedAt` set).
3. Registry upload + dedup-on-ingest: an Upload button on DocumentsView calling the existing `saveDoc`; before write, look up `contentHash` and offer "already in vault ‚Äî link instead." (~1 day.)
4. ## 7. AI assessment

Local/BYO-key: AIPanel ‚Üí `runChat.streamText` with the full tool belt (`tools.ts`: 8 read-only, 11 mutating), provider per settings (Ollama URL / OpenAI / Anthropic direct-from-browser with the dangerous-direct-access header; keys session-memory only, never persisted ‚Äî `settings-store.ts:77,147-175`). **No project context packet is sent in local mode at all** ‚Äî context arrives via read-only tool calls (`runChat.ts:65-74`). Every mutating tool's `execute` is swapped at module load for "queue an approval proposal" (`tools.ts:1002-1016`); the real executor runs only from `approveAIProposal` after a user click, behind a re-checked engine dry-run preview (`approval-preview.ts` runs the actual `execute*` engine functions), a fail-closed undo snapshot, and the same store mutators the UI uses ‚Äî which journal into the title ledger with synchronous parity (`workspace-store.ts:438-453` ‚Üí `title-action-log.ts:520-574`). Hosted: no tools in the request at all; context = `buildAIAppContext` (minimal counts-only by default ‚Äî verified clean of names/fractions/ids ‚Äî full mode gated behind per-workspace, non-persisted disclosure), through an Amplify rewrite to a Lambda Function URL (auth NONE, in-handler Cognito JWT verify), strict v1 schema (model/stream/messages only, everything else 400), model hardcoded `gpt-4o-mini`, 2048 max tokens, daily 500k-token DynamoDB ceiling, content never logged. This is a sound architecture. The wiring quality issues are DA-M3 (provenance flattened ‚Äî the gate assertion never sees `origin:'ai'` live), DA-M12 (stale preview render + snapshot TOCTOU), DA-M13 (rate limiting), DA-L3/L4/L5 (dead code, hosted wizard 400, README drift), and `AIPanel` never trimming to the proxy's 50-message cap.

### (b) Safety findings

- **Approval bypass: none found.** Mutate-then-ask is structurally impossible (execute swapped at module load, both modes); hosted mutating surface is absent twice (no tools sent; proxy rejects `tools`/`tool_choice`). `deleteNode` is leaf-only with the cascade refusal hard-coded (`tools.ts:713-736`).
- **Exfiltration: no model-reachable channel.** No tool does network/file/clipboard I/O; AI output renders as plain React text ‚Äî no markdown/HTML, no image-beacon channel (`AIPanel.tsx:868-899`); no `dangerouslySetInnerHTML` in src.
- **Prompt injection: surface exists, blast radius is the approval click.** Imported CSV text persisted into `remarks`/`landDesc` re-enters the model via tool results (truncated) and hosted-full context (untruncated, `app-context.ts:461`). A hostile cell can steer the model into queueing plausible proposals; the engine-computed typed preview is the real defense. Residual risk = approval fatigue. Recommendations: cap `remarks` in context (`cleanText` length cap, one line), and label proposal cards whose triggering turn contained imported text ("references imported document text") so the operator's stop-and-ask instinct fires (the backlog's LLA-M10 staging path already does the heavy half).
- **Ledger interaction: the real problems are not AI-specific** ‚Äî they are DA-H2 (undo erases the ledger ‚Äî triggered from the AI panel), DA-C1 (unjournaled actions an AI tool doesn't call today, but `createDeskMap` with `initialNodeIds` is one journal-free path an AI tool touches with `[]` ‚Äî keep it pinned by test), and DA-M3 (provenance).
- **Keys/tokens:** cloud keys session-only (good); Cognito ID token in localStorage (`auth/session.ts:24-43`) is the standard-SPA weak link ‚Äî acceptable now, revisit if hosting broadens (httpOnly-cookie BFF is the eventual answer, not worth it for a single operator).

### (c) Where AI should go ‚Äî integration contract

The roadmap's direction (deterministic tools, hybrid retrieval, CitationVerifier, inline ask, Cmd+K) is right, and more of it exists than the roadmap admits: `verifyCitationSupport` is implemented with anchor-grade checks (`projections.ts:330-417`), extraction-run invariants are enforced (`extraction-runs.ts:165-186`), and the AI mutation guard is machine-checked (`ai-mutation-guard.ts:19-46`). Cheap (the data structures exist), kills the highest-likelihood AI failure (confident wrong arithmetic near the math), and is exactly the "AI builds what it can support and stops" posture the operator already enforces.
