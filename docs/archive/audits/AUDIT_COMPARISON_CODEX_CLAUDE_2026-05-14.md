## Disagreements after re-examination

### Root total should not be hard-blocked in `createRootNode` without a workflow change

Claude is correct that `executeCreateRootNode` checks the individual root fraction only (`src/engine/math-engine.ts:483`, `src/engine/math-engine.ts:488`) and returns success without `assertRootTotalNotWorsened` (`src/engine/math-engine.ts:523`, `src/engine/math-engine.ts:539`). However, the current product model treats over-100 mineral coverage as a warning-only reconciliation state (`LANDMAN-MATH-REFERENCE.md:25`, `LANDMAN-MATH-REFERENCE.md:26`), and Desk Map tests assert `1.5` current ownership with `-0.5` missing ownership for temporary over-100 coverage (`src/components/deskmap/__tests__/deskmap-coverage.test.ts:592`, `src/components/deskmap/__tests__/deskmap-coverage.test.ts:608`, `src/components/deskmap/__tests__/deskmap-coverage.test.ts:609`). The AI import flow also tells the assistant to create standalone roots first and graft later (`src/ai/AIPanel.tsx:168`, `src/ai/system-prompt.ts:34`, `src/ai/system-prompt.ts:52`).

Reconciled action: add clearer warning/finalization UX and tests around over-100 root creation; do not hard-block `createRootNode` unless the import/reconciliation workflow changes.

### NPRI discrepancy handling is UI-warning behavior, not missing validation by default

Claude is correct that `findNpriBranchDiscrepancies` is not invoked by engine executors (`src/engine/math-engine.ts:996`), but it is invoked by Desk Map highlight state (`src/views/DeskMapView.tsx:353`, `src/views/DeskMapView.tsx:356`) and fed to tree rendering (`src/views/DeskMapView.tsx:1039`, `src/views/DeskMapView.tsx:1041`, `src/views/DeskMapView.tsx:1044`). Since NPRI over-claims are documented as warning-only (`LANDMAN-MATH-REFERENCE.md:25`), I would not rank this as a correctness blocker unless the product wants executor-level warnings returned consistently after every mutation.

### Proxy-side read-only enforcement needs precise wording

Claude's "hosted-mode mutation block is client-side only" is true, but it can sound like the Lambda can mutate LANDroid state. It cannot; LANDroid mutations happen in the browser through local Zustand/Dexie state. The more precise risk is that the hosted proxy does not enforce an allowlisted OpenAI request shape, so a malicious or modified client with a valid ID token can use the server-held OpenAI key for requests outside the intended hosted tool policy (`src/ai/runChat.ts:60`, `backend/ai-proxy/src/request-policy.ts:57`). This should still be fixed, but the concrete app-state mutation bug remains `setActiveDeskMap` leaking through the hosted read-only subset (`src/ai/tools.ts:847`, `src/ai/tools.ts:899`, `src/ai/tools.ts:909`).
