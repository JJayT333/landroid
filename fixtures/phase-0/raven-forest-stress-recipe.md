# Raven Forest Stress Fixture Recipe

W2 is a rebuild stress-test shape, not a committed exact workspace export.

The purpose is to let Phase 0.5 and later rebuild phases generate a Raven
Forest-sized project on demand, using the same behavioral stress points each
time, without freezing today's exact seed as a permanent product fixture.

## Current Reference Source

Today, the closest source is `buildCombinatorialWorkspaceData()` in
`src/storage/seed-test-data.ts`.

Observed shape on the Phase 0 branch:

- 10 tracts
- 2 units
- 1,476 title nodes
- 658 owner records
- 414 lease records
- 145 PDF mappings
- 3 WI assignments
- 6 ORRIs
- 5 federal lease records through the federal lease seed

This source is acceptable as the current W2 reference for Phase 0 planning, but
the rebuild should generate its own deterministic stress fixture rather than
depending forever on this exact app seed.

## Rebuild Stress-Test Requirements

A rebuild-era Raven Forest stress fixture should be generated by a script with
a fixed seed and explicit scale parameters. It should produce the same category
of pressure as Raven Forest, not the same fictional facts.

Minimum shape:

- 1,000-3,000 title nodes
- 10-30 tracts
- 2-5 units
- 200-1,000 document records using deterministic stub PDFs
- 200+ owner records
- 100+ lease records
- mixed mineral, NPRI, ORRI, WI assignment, and retained-WI cases
- at least 5 federal leasing reference records that do not affect Texas math
- repeated owners across tracts and units
- overlapping leases with deterministic tie-break expectations
- malformed or warning-producing records for strict parse and warning behavior
- one or more over-100% mineral coverage scenarios
- at least one unit-focus transfer-order case

## Determinism Rules

The generator must use:

- fixed seed input
- fixed timestamps
- stable IDs
- deterministic sort order
- deterministic stub PDF bytes
- deterministic content hashes
- no real title documents
- no browser-only APIs unless the test harness provides deterministic shims

The generator should emit a manifest with:

- seed value
- generator version
- generated timestamp
- node count
- tract count
- unit count
- owner count
- lease count
- document count
- expected warning counts
- expected performance-baseline workload IDs

## Artifact Policy

Do not commit a large `.landroid` export for W2 by default.

Preferred rebuild policy:

- commit the generator
- commit a small manifest
- commit small golden summaries where useful
- regenerate the full stress workspace during baseline capture or CI jobs that
  explicitly opt into stress fixtures

Commit the full generated `.landroid` only if it stays small enough to review
and diff reasonably after stub PDFs are used.

## Baseline Uses

Use W2 for:

- Desk Map first render and interaction timing
- workspace open/load timing
- autosave serialization timing
- document registry first render timing
- packet preview timing
- `.landroid` export/import timing
- Leasehold transfer-order computation timing
- Flowchart print/page layout timing
- iPad Pro-class PWA acceptance checks

## Non-Goals

W2 should not:

- encode real client facts
- include real lease forms or title documents
- replace W1 Vulcan Mesa for small golden-master behavior checks
- become the source of truth for product behavior
- hide current behavior changes behind randomized fixture drift
